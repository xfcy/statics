  3. 所有的系统为 B/S 结构，也就是说最终的表现形式是一个网站

  4. 毕业设计本身包含设计和开发两部分

    • 设计优先于开发进行
    • 采用面向对象的软件设计，也就是说至少要对系统进行 UML 建模
    • 对于开题报告、中期报告和最终的毕设论文，群里会有示例模板，但是请不要只修改题目和内容中的关键词，这样肯定过不了我这关的。范文只用来参考文章组织结构，内容请自己撰写

  5. 大部分童鞋的系统属于信息管理系统

    • 这种系统的总体功能就是用来管理某一类信息

      • 每种系统的主题不同，所管理的信息不同，逻辑不同
      • 对信息的分类也是这种系统需要重点设计的部分

    • 往往包含某几个类型的用户角色，比如普通用户、管理员

      • 这种用户系统设计建议采用 RBAC 形式的权限管理，更为灵活
      • 当然 RBAC 也比较复杂，如果不能接受可以自行设计用户权限系统
      • 但是不要只用一个字段来区分用户和管理员

    • 所有题目共同包含的模块为用户模块，详细说明如下

      • 用户

        • 用户拥有的信息

          • 用户名（邮箱、手机号或其他形式的用户名，登录时使用，唯一，不可更改）
          • 密码（千万不要明文存储，哪怕做一下 md5 呢，当然 bcrypt 更好）
          • 昵称（显示在用户界面上的名字，可以自定义，不一定唯一）
          • 其他和具体题目相关的用户信息
          • 其他的信息比如头像啊什么的自行发挥，有无皆可

        • 用户拥有的操作

          • 用户的登录注册属于必须有的操作，但是注意，有管理员的系统中，管理员是不允许直接注册的
          • 用户的登录状态如何维护（不用太过复杂，老师甚至不会注意，但是注意一下会比较好
          • 用户信息的维护，包括昵称修改、密码修改等等
          • 其余的都是和具体题目的业务相关了

      • 管理员

        • 拥有系统中其他用户角色的所有权限
        • 可以更改用户的用户组
        • 可以更改用户组的权限
        • 可以修改任意用户信息
        • 不可通过注册直接成为管理员
        • 系统管理员需要有删除不良信息的权限

      • 用户组（建议）

        • 建议使用用户组来区分用户权限
        • 基于 RBAC 模式更好
        • 当然只是建议

  6. 包含在开发工作量中的内容有

    • 前端：就是用户直接可见的网页

      • 前端要求采用html5标准，其余不限
      • 也就是说做出的网页在常用现代浏览器上可以正常运行
      • 比如 Google Chrome、Firefox 、Microsoft Edge 等
      • JavaScript框架如 Vue、React、Angular 等没有什么要求，用 jQuery 都行
      • 建议选用前端开发框架，可以让前端页面不那么丑

    • 服务端：提供数据接口（前后分离式的开发），或者渲染静态页面（嵌入模板式开发）

      • 后端语言不限，可以选用自己顺手的方式来开发
      • 但是尽量采用常见的开发方式，比如 Java/PHP/Python 等，避免出现答辩老师不熟的技术被卡
      • 建议选择一个开发框架来规范代码并加速开发，比如php的CodeIgniter/Laravel、Java的Spring Boot、Python 的 flask 之类的

    • 数据库

      • 关系型数据库尽量选用 MySQL 吧

        • 别的也行，但是准备好被问为什么不用MySQL这样的问题
        • 关系型数据库必须使用，就算满足下一条提到的条件，也必须在一些业务中采用关系型数据库

      • 对于个别需要特殊存储结构的题目，可以采用其他 NoSQL 的数据库，在写论文的时候最好写清楚为什么采用

        • 文档数据库 MongoDB
        • 图数据库 Neo4j
        • 内存K-V存储 Redis 和 Memcached （一般做缓存用）

  7. 我们的毕业设计不要求大家做高可用高性能，但是起码要能用，无明显bug，最终完成后老师会查看你们的论文和代码，不合格的话无法参加答辩，事关毕业请务必认真对待

*基于百度地图的城市路灯监控管理系统

  • 需要实现的功能

    • 用户和用户组的功能（见第一条

      • 巡查人员

        • 不同的巡查人员有不同的巡查线路
        • 巡查人员发现路灯状态异常后可上报
        • 异常状态的路灯完成维修后需要巡查员确认

      • 维修人员

        • 不同维修人员有自己负责的线路
        • 根据巡查人员提交的异常信息进行维修
        • 维修完成后提交维修信息，供巡查人员确认

      • 调度人员

        • 负责管理路灯信息
        • 给巡查人员和维修人员设置负责的线路

      • 管理员

    • 路灯管理

      • 首先肯定要有路灯的基本信息（路灯型号、编号、所在的道路、经纬度、状态等
      • 添加、修改、删除
      • 路灯要显示在地图上，用不同的标识表示路灯的状态

    • 通知系统

      • 维修人员登录系统后，需要看到自己负责的线路的故障信息
      • 巡查人员可以看到自己还有哪些没有巡查，以及维修人员提交的维修信息
      • 调度人员可以看到所有的异常信息、维修信息等，并可以指派人员去维修或验收
      • 管理员不会收到通知，但可以主动查看以上所有信息

  • 难点

    • 百度地图的调用、路灯的表示

      • 可以申请百度的key，来调用百度的接口
      • 或者使用 openlayers 这样的开源地图工具，加载百度的瓦片图来实现

*通用工作任务审批与督查督办系统

  • 这个题目基于一个真实的项目，具体要求如下

    • 简单来说，这是一个工作流系统，因此可以考虑采用各种工作流引擎

    • 在企业工作中有各种各样的任务，这些任务来自于各个部门的各种需求

    • 这些任务一般需要由具体的负责人员提出，并报请上级批准，每一级审批人员可以决定是否向自己的上级转发该任务，也就是任务流转

      • 任务需要有起止时间、任务内容、目标、负责人（一般是提出任务的申请人）、附件等

    • 在所有层级审批完成后任务状态变更为执行中，所有审批过该任务的人员可以查看该任务的状态，成为该任务的监督人

    • 任务的负责人可以（需要）定期更新任务的执行状态

    • 任务的监督人可以对项目进行催办，并填写相应理由

    • 任务完成后，由负责人提出办结申请，并由监督人进行审核（一级一级的审核，和任务开始时的审批流程相同

    • 在任务中转的每一步，都需要使用站内通知的方式，将具体信息通知到相关人员

    • 以上每一步的用户操作都需要有操作记录，并在任务详情页面可见

*基于Web的高校在线考试与评阅系统

  • 要实现的功能

    • 用户模块

      • 学生

        • 学生的角色很简单，属于某个班级，可以查看自己的考试安排，按时参加考试，查看考试结果即可

      • 教师

        • 每个教师有自己的学科
        • 可以建立题库，添加题目
        • 可以从题库中挑选题目，组成试卷
        • 可以给自己任教的班级添加考试，并指定使用的试卷
        • 考试截至后，可以查看所有学生的试卷

      • 管理员

    • 题库、题目和试卷

      • 题库由教师建立，并向其中添加题目
      • 试卷由教师选择题库中的题库来创建
      • 题库和试卷均有学科属性，不同学科的题目无法添加进题库，也无法组成试卷

    • 考试模块

      • 学生在教师指定的时间，作答教师指定的试卷，超时则无法作答
      • 考试中需要有一定的反作弊措施，比如切出试卷等操作需要记录，教师评阅时展示出来

    • 评阅模块

      • 系统可以根据题库中设置的答案，自动评阅选择题
      • 可选：根据教师设置的答案关键词自动批阅填空题、简答题
      • 主观题展示给教师，由教师进行批改

*基于网络爬虫的新闻订阅抓取系统

  • 这个题目的重点在于爬虫

    • 目前大部分门户网站都有反爬系统，但是在该题目中绕过反爬不是重点（而且强行绕过反爬规则属于违法行为
    • 爬虫的行为需要控制，一旦大规模爬取数据造成 DDOS，也属于违法行为
    • 所以就找三个好怕爬的网站，爬一些新闻信息即可

  • 具体要求

    • 用户根据自己的兴趣，设置爬取关键词（比如“特朗普”）和类别（比如“社会新闻”，可以自己给出选项让用户选择）

    • 爬虫定时根据用户设置的信息去爬取几个网站上的新闻

      • 对于关键词，使用标题匹配即可
      • 对于类别，直接全部抓取就行

    • 可以不存储新闻内容，在用户点开新闻的时候再去爬取相应页面的具体内容

  • 在该系统中，爬虫的规则编写属于难点，其余部分和常规的管理系统没啥区别

    • 对于定时爬取，考虑使用Linux系统的Crontab功能，为此又需要使用任务队列，若有其他想法可与我交流
    • 对于新闻内容的爬取，可以找一些有RSS源的网站，RSS属于结构化的信息，且本身就是提供给用户订阅的接口，比较好爬（不一定每个网站都有RSS，为了工作量，尽量不要都使用该方法
